BotOps Control Plane — 通用化完整頁面架構盤點

目的：將原加密貨幣交易所面試專案改造為通用型 CS Bot 營運平台——平台層通用、產業層以 domain pack 模組組織；示範資料沿用 crypto／fintech 場景作為 mock data，UI 不做 pack 切換器。本文件盤點全部三層頁面結構、每頁功能、使用情境與邊界狀態，供確認後執行。

層級定義 — L0：全域框架 ／ L1：主導覽模組（頁面）／ L2：頁內分頁、區塊 ／ L3：詳情面板、彈窗、抽屜與其中的操作。

保留 現有功能不動 調整 去產業綁定／改設定驅動 新增 通用化新功能
目錄
  1. L0 全域框架（Shell）
  2. Dashboard 營運總覽

L0 · 全域框架（Shell）

所有頁面共用的導覽與全域狀態。

結構與功能

  • 側邊導覽列：六個 L1 模組入口 保留（新增 Agent Assist、Settings 入口 新增）
  • 工作區標題列：顯示目前頁面標題 保留
  • 語系切換：EN ／ zh-TW，localStorage 持久化 保留
  • Domain pack 僅存在於資料層：mock data 收攏為 packs/crypto 模組，UI 不顯示切換器；未來換產業＝換 pack 檔，不動平台層程式 新增
  • 響應式：窄螢幕收合為抽屜導覽；表格 reflow 成 label／value 卡片 保留

狀態與邊界

  • 出貨 pack：Crypto／Fintech（加密貨幣交易所類客服場景作為 mock data，KYC、Security-L2 等術語屬 pack 內容，予以保留）

1 · Dashboard 營運總覽

頁面定位：PM 每天的起點。30 秒內回答「bot 現在健康嗎？今天該先處理什麼？」

主要使用者：AI Chatbot PM、Bot Ops、Support Lead

L2 · 區塊

  • 1.1 Priority queue 優先處理佇列 保留
    • 項目類型：被阻擋的發布套件、SLA 風險工單、待重建索引的知識、未結案 badcase、待審 trace
    • 每項欄位：嚴重度、負責人、到期時間、原因摘要、CTA（點擊直達對應模組）
  • 1.2 營運 KPI 卡片區 保留
    • 對話總量／已審查互動數、自動解決品質、交接與工單轉換率、知識庫規模、進行中 bot 工單數、知識缺口數、badcase 趨勢、發布就緒度
  • 1.3 營運經濟學卡片 新增
    • 每張自動解決工單成本 vs 人工處理成本
    • 本期 deflection 折算節省人力時數
    • badcase 修復的量化回收（修復前後該案例簇的人工處理量差）
  • 1.4 Top questions ／ 重複進線熱點 保留
  • 1.5 發布門檻速覽：解釋發布就緒度，不主導版面 保留

L3 · 詳情與操作

  • Priority queue 項目詳情：發生了什麼、為什麼重要、下一步做什麼（三問標準）保留
  • KPI 卡片點擊 → 跳轉 Quality & Release 的 KPI 分頁並帶入對應指標 調整
  • 經濟學卡片點擊 → 假設參數面板（人工單位成本、平均處理時長可調，展示計算透明度）新增

狀態與邊界

  • 佇列全空（一切健康）：顯示正向空狀態，不留白
  • 資料未同步：顯示最後同步時間與重新整理入口

核心情境

  • PM 早上巡檢：先看 priority queue，再掃 KPI 異動，10 分鐘決定今天的工作排序
  • 事故爆量：多通路同類訊號激增 → queue 出現 incident 候選 → 直達 Conversations 聚類檢視
  • 發布前檢查：從發布就緒度卡片直達 Release gates
  • 季度匯報：用經濟學卡片的節省人時／成本數字對上級說明 bot 投資報酬

2 · Conversations 對話與訊號審查

頁面定位：回覆後（post-reply）的品質問責。不是客服回覆台，是 PM 的 QA 審查與 eval 素材來源。

主要使用者：PM、Bot Ops

L2 · 區塊

  • 2.1 Intake 多通路訊號列表 保留
    • 通路篩選：Web/App Chat、X、LINE、Telegram、Discord、內部回報 保留
    • 欄位：回報者類型（客戶／社群版主／內部／法遵／公開貼文）、來源信任度、重複叢集 ID、地區、語言、產品線、優先級
  • 2.2 互動審查區（Chat + Trace） 保留
    • 對話串：user／assistant／system 訊息、穩定訊息 ID、引用標記
    • Response trace 時間軸：來源正規化 → 意圖 → 風險 → 改寫 → 檢索 → 生成 → 引用 → 驗證 → 交接，每節點含狀態（pass／watch／blocked）與輸入輸出參照
    • 引用證據面板：命中的 KB chunk 高亮顯示

L3 · 詳情與操作

  • 審查 live trace（寫入 audit log）保留
  • 另存為 eval case：打包訊息、引用、trace、來源與預期行為 保留
  • Trace 節點抽屜：單一節點的完整輸入輸出與診斷 新增（原 PRD next batch）
  • 跳轉：引用 chunk → Knowledge 對應文件；交接場景 → Tickets 的 handoff preview 保留
  • 叢集檢視：同 duplicateClusterId 的訊號聚合顯示，判斷是否升級為 incident 新增

狀態與邊界

  • 篩選後無訊號：空狀態附建議動作
  • trace 不完整（某節點缺漏）：明確標示缺漏而非留白
  • 低信任來源（公開社群貼文）：顯著標記，提示查證後再採信

核心情境（crypto mock pack）

  • 跨境支付政策申訴、帳號盜用轉帳凍結、身分驗證（KYC）被拒、轉帳延遲爆量
  • 共通情境（跨產業皆成立，面試時可口頭映射到對方領域）：KB 缺漏導致胡答、高風險問題未轉人工、多通路同題訊號需要聚類判斷

3 · Knowledge 知識治理

頁面定位：bot 回答的「根據」從哪來、品質如何、缺口在哪。FAQ／SOP／RAG 的完整生命週期管理。

主要使用者：Knowledge Owner、PM

L2 · 分頁

  • 3.1 知識清冊（預設分頁） 保留
    • 總覽指標：已索引文件數、待重建索引數、chunk 總數、目前檢索設定
    • 表格欄位：ID、標題、分類、標籤（地區／風險等級）、查詢量、生命週期狀態（Draft／Review／Published／Archived）、索引狀態、操作
    • 檢索政策面板：KB snapshot 版本、索引政策、引用政策（僅 citation-allowed chunk 可支撐回答）
  • 3.2 Gap mining 知識缺口挖掘 新增
    • 新興問題簇列表：以未解決＋交接對話聚類，欄位：簇名、量體、週趨勢、樣本語句、關聯 KB（缺漏／薄弱）
    • FAQ 候選工作流：AI 起草答案草稿 → Knowledge Owner 審核（採納入庫／退回重寫／標記不適合自動化）
    • 成效追蹤：候選入庫後該簇的 deflection 變化（before／after）
  • 3.3 SOP 管理 新增
    • SOP 結構化檢視：步驟、分支條件、觸發情境、對應的 bot 行為（可自動執行／需人工確認／禁止自動化）
    • SOP 與 chat flow 節點的對應關係表

L3 · 詳情與操作

  • 知識紀錄詳情：負責單位、語言、生效日、向量索引、檢索設定、最後索引時間、版本歷程 保留
  • Chunk 證據面板：分段內文、token 數、citation 許可 保留
  • 操作（皆有可見回饋）：檢視 chunks、審查引用範圍、排入重建索引 保留
  • Chunk 索引工作紀錄（job log）新增（原 PRD next batch）
  • Gap 候選詳情：樣本對話、AI 草稿、審核意見欄、決策按鈕 新增

狀態與邊界

  • 文件已發布但未索引 → 顯著警示（回答無法引用）
  • 文件過期（effectiveTo 已過）→ 自動標示待下架
  • 高查詢量＋低引用成功率的文件 → 標記為優先改寫對象
  • Gap 簇樣本不足（< 門檻值）→ 顯示為觀察中，不產草稿

核心情境

  • Knowledge Owner 週巡檢：從表格排序找出高查詢、低品質、待重建索引的文件
  • 政策更新：改文件 → 走 Review → Published → 重建索引 → 確認新回答引用新版本
  • Gap 審核例會：過一遍本週新興問題簇，決定哪些寫 FAQ、哪些改 SOP、哪些保持人工
  • 發布前 KB snapshot 凍結：確認 eval 用的 snapshot 與候選發布一致

4 · Agent Assist 座席輔助治理 新模組

頁面定位：管理「AI 幫人工座席」這條產品線的品質——建議回覆與 AI 摘要好不好，用座席的實際行為來量。不是回覆台，是治理面板。

主要使用者：PM、Support Lead

L2 · 區塊

  • 4.1 指標總覽卡片 新增
    • 建議回覆採納率（直接採納／修改後採納／棄用）
    • 修改幅度分布（edit distance 分桶）
    • AI case summary 重寫率
    • 採納 vs 未採納的平均處理時長差
  • 4.2 建議回覆明細表 新增
    • 欄位：案例、queue、建議內容摘要、座席動作、修改幅度、處理時長
    • 篩選：queue、意圖類型、採納狀態、時間區間
  • 4.3 Summary 品質比對 新增
    • AI 摘要 vs 座席最終摘要的並排 diff
    • 常見改寫模式歸納（漏了什麼、多了什麼、語氣問題）

L3 · 詳情與操作

  • 建議詳情抽屜：完整建議內容、座席最終版本 diff、關聯 trace 連結 新增
  • 標記動作：將低品質建議標記為 badcase（流入 Quality 的 badcase 迴路）新增
  • 政策建議：對高採納率＋低風險的意圖，提示「可評估開放自動送出」；對高棄用率意圖提示「檢討 prompt 或關閉建議」新增

狀態與邊界

  • 某 queue 未開通 assist：空狀態說明開通條件
  • 樣本量不足的指標：顯示信賴區間警示，避免小樣本誤導

核心情境

  • 新 prompt 版本上線後，比較採納率與修改幅度是否改善
  • 找出建議品質最差的 queue，決定優先修哪類意圖
  • 用「摘要重寫率下降」向 CS Ops 證明 AI 摘要值得信任
  • 決策哪些意圖從「建議」升級為「自動送出」——採納率、風險等級、修改幅度三條件同時滿足

5 · Tickets & Handoff 工單與交接

頁面定位：bot 建立的工單消耗了多少人工容量、交接品質好不好。PM 的治理協調視角，不是座席的工作終點。

主要使用者：Support Lead、PM、各 queue 負責單位

L2 · 區塊

  • 5.1 工單表格 保留
    • 欄位：工單 ID、問題摘要＋下一步、queue、優先級、SLA、處理狀態、PIC、操作
    • Queue 與 PIC 沿用 crypto mock 現值（KYC Review、Security-L2 等），型別定義移入 pack 模組 調整
    • 篩選：queue、優先級、狀態、SLA 風險 新增（現況無篩選）
  • 5.2 Queue health 風險摘要：高優先數、已升級數、SLA 關注數 保留
  • 5.3 Handoff preview 交接包預覽 保留
    • 交接原因、目的地 queue、必填欄位清單、AI 摘要、風險警語

L3 · 詳情與操作

  • 工單詳情：queue、狀態、PIC、SLA 到期、來源訊號、trace 參照、AI case summary、下一步、處理時間軸 保留
  • 操作（皆有可見回饋、本地持久化）：改狀態（Open／Pending review／Escalated／Resolved）、改 PIC 保留
  • 交接品質標記：交接包欄位不完整 → 標記回 badcase 迴路 新增

狀態與邊界

  • SLA 已逾期：紅色標示並浮到 Dashboard priority queue
  • 同一事故產生的工單簇：以叢集標記聚合，避免逐張處理
  • 交接必填欄位缺漏：明確列出缺什麼

核心情境

  • Support Lead 晨會：按 queue 與 SLA 排序分派 PIC
  • SLA 逾期升級：狀態改 Escalated，追蹤升級後的處理時間軸
  • PM 月度回顧：bot 建單量與人工容量消耗趨勢，決定哪類交接該回頭改 bot
  • 交接包品質抽查：高風險 queue 的 handoff preview 是否欄位齊全、風險警語正確

6 · Quality & Release 品質與發布

頁面定位：整個產品的縱深所在——用 eval 證據守住發布，用 badcase 驅動迭代，用 audit 保證可回溯。五個分頁漸進揭露。

主要使用者：PM、Bot Ops、Compliance

L2 · 分頁一：Release gates 發布門檻（預設）

  • 發布套件列表：flow／prompt／KB snapshot／檢索設定／judge 五版本組合、關聯 eval run、退化數、狀態（Draft → Eval running → Blocked／Compliance review → Canary simulation → Ready → Rolled back）保留
  • 決策操作（寫入 audit）：Promote／Block／Request review，附決策理由欄 保留
  • Flow version diff 檢視 新增
    • 候選 vs 現行版本的差異：改了哪些 flow 節點、哪段 prompt、換了哪個 KB snapshot、檢索參數異動
    • 唯讀流程拓撲圖＋變更高亮（不做拖拉編輯器）
  • 發布核准歷程 新增（原 PRD next batch）

L2 · 分頁二：KPI

  • 指標卡片：自動解決率、交接率、重複進線率、引用失敗率、SLA 風險，各含目標值、趨勢與洞察 保留
  • 分群下鑽表：來源通路 × 案例類型 保留
  • KPI → 案例穿透：點擊異常分群 → 直達 Conversations 帶入該分群篩選，從指標走到真實對話 新增
  • 語言 × 地區品質對等矩陣：熱力圖標出各語言／地區的自動解決與引用失敗落差 新增

L2 · 分頁三：Eval runs 評測

  • 評測批次表：整體分、引用支持率、交接安全召回、高風險自動回答率、退化數 保留
  • 批次詳情：資料集、五版本組合、七維度分數（意圖、來源正規化、檢索、引用、可行動性、風險交接、可追溯）保留
  • 操作：啟動評測、CSV 匯出（含去識別化預覽彈窗）、連結已存 eval case 保留
  • Judge 校準卡 新增
    • LLM judge 與人工標註一致率、抽樣複核佇列、judge 換版漂移警示
  • Eval case 詳情檢視：單筆案例的輸入、預期來源、預期行為、禁止行為、must-handoff 標記 新增（現況只有列表計數）

L2 · 分頁四：Badcases 失敗案例

  • 列表＋狀態控制（Open／In review／Fixed）保留
  • 詳情彈窗：低分維度、trace 診斷、需調整的鏈路節點、調整方式、重測指標、預期分數變化、負責單位 保留
  • Before／After 對照：修復後重測結果與原 badcase 並排 新增
  • 來源擴充：badcase 可由 Conversations 審查、Agent Assist 標記、交接品質抽查三處流入 新增

L2 · 分頁五：Audit 稽核

  • 事件流：trace 審查、eval case 儲存、評測啟動／完成、CSV 匯出、發布決策，含 actor、實體參照、時間 保留
  • 篩選：事件類型、actor、時間區間 新增（現況無篩選）
  • 事件詳情：三問標準（發生了什麼／為什麼重要／下一步）保留

狀態與邊界

  • 套件被門檻阻擋：Promote 按鈕不可用，明列失敗的門檻項
  • 評測執行中：runner 狀態即時顯示，完成前不可再次啟動
  • Judge 一致率低於門檻：eval 結果標示「待人工複核」，不得作為發布依據
  • 匯出僅含 run-level 指標，不含客戶識別資訊（隱私邊界聲明保留）

核心情境

  • 發布決策會議：看 flow diff 確認改了什麼 → 看 eval 對照 → 看退化數 → Promote 或 Block，理由入 audit
  • 退化攔截：候選版本某維度分數下降 → 自動 Blocked → 從 badcase 找根因
  • KPI 異常追查：引用失敗率上升 → 穿透到對話 → 發現某文件過期 → 回 Knowledge 修復
  • 稽核回溯：三個月前的發布決策依據什麼 eval 證據、誰核准的，五分鐘內調出完整鏈

7 · Settings 設定 新模組

頁面定位：平台組態的唯讀展示入口，呈現風險政策與治理邊界（domain pack 不在此顯示，僅為資料層模組）。

L2 · 區塊

  • 7.1 平台組態（唯讀展示） 新增
    • 風險政策：哪些意圖禁止自動回答、必須交接的條件
    • SLA 規則、eval 門檻值、judge 版本
  • 7.2 RBAC 角色模型（roadmap 預告，唯讀） 新增
    • Viewer／Operator／PM Admin／Compliance 四角色權限矩陣（對應 production readiness plan，展示而不實作）
