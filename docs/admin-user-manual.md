# BotOps Control Plane 使用手冊（v2.0）

## 產品定位

BotOps Control Plane 是給 AI Chatbot PM 與客服營運團隊使用的治理平台，不是一線客服的回覆台。Bot 已經即時回覆客戶；這個平台處理的是回覆後（post-reply）的品質問責：審查 bot 說了什麼、根據是什麼、缺口在哪、座席輔助好不好用、發布該不該放行，以及每個治理動作是否留下紀錄。

平台層通用，示範資料採 crypto domain pack：介面中出現的 KYC、Security-L2、Travel Rule、跨境支付等術語都是示範場景內容，不是平台功能的一部分。換一個產業只需要換資料層的 pack，模組與治理機制不變。

## 使用者角色

- **PM（AI Chatbot PM）**：每天的巡檢起點，負責審查 trace、決定發布、追 KPI 異常。
- **Bot Ops**：跑評測、診斷 trace、修 badcase、驗證 prompt 迭代。
- **Knowledge Owner**：維護知識清冊、審核缺口挖掘的 FAQ 候選、管理 SOP。
- **Support Lead**：看工單隊列與 SLA、抽查交接包、檢視座席輔助成效。

## 操作前準備

1. 啟動本機產品：

```bash
npm run dev
```

2. 開啟：

```text
http://127.0.0.1:5173/
```

3. 右上角可切換 EN／繁中，選擇會存在 localStorage，下次開啟沿用。
4. 左側導覽共六個模組：`儀表板`、`對話審查`、`知識治理`、`座席輔助`、`工單與交接`、`品質與發布`。每個模組標題下方有一行任務說明，寫明這個頁面用來完成什麼。建議先看儀表板掌握整體狀態，再進入各模組。
5. 詳情類內容（知識文件、工單、座席建議、缺口簇、SOP）一律以右側抽屜（drawer）開啟，手機上為全螢幕；按 `Esc`、點抽屜外的遮罩或右上角 X 都可以關閉。

## 各角色快速上手

### PM 最常用的 3 個任務

1. **早上巡檢**：開 `儀表板`，先看優先工作清單（阻擋發布、SLA 風險工單、待重建索引），再掃 KPI 卡，點各項 CTA 直達對應模組。
2. **審查互動並轉存評測案例**：`對話審查` → 點 `檢視回覆與紀錄` → 檢查回答與 trace → `建立評測案例`。
3. **發布決策**：`品質與發布` → `發布門檻` → 點 `檢視變更` 看 flow diff → 確認門檻結果 → `推進`／`阻擋發布`／`請求審查`。

### Bot Ops 最常用的 3 個任務

1. **跑評測**：`品質與發布` → `評測批次` → `啟動評測`，確認 runner 狀態變為已完成，比較各批次分數。
2. **修 badcase**：`品質與發布` → `失敗案例`，從詳情看低分項目與需調整節點，處理後把狀態改為 `Fixed`。
3. **驗證 prompt 迭代**：`座席輔助` 看新版本上線後的採納率與修改幅度是否改善。

### Knowledge Owner 最常用的 3 個任務

1. **週巡檢知識清冊**：`知識治理` → `知識清冊`，找出需重建索引的文件，點 `排入重建索引`。
2. **審核 FAQ 候選**：`知識治理` → `缺口挖掘`，逐簇審核 AI 草稿，決定採納入庫、退回重寫或標記不適合自動化。
3. **確認 SOP 自動化邊界**：`知識治理` → `SOP 管理`，檢查每個步驟的邊界標記是否正確，特別是禁止自動化的步驟。

### Support Lead 最常用的 3 個任務

1. **晨會分派**：`工單與交接` 看隊列狀態摘要（高優先、已升級、SLA 關注），調整工單狀態與負責人（PIC）。
2. **抽查交接包**：點開一張工單的詳情抽屜，抽屜下半部的人工交接預覽顯示該工單的交接包，確認觸發原因、必要欄位與風險警示齊全。
3. **檢視座席效率**：`座席輔助` 看採納率與處理時長差，判斷 AI 輔助對座席是幫忙還是干擾。

## 模組操作

### 1. 儀表板

**你想完成什麼**：30 秒內回答「bot 現在健康嗎？今天該先處理什麼？」，並把 bot 價值翻譯成成本語言。

怎麼做：

1. 點擊左側 `儀表板`。
2. 從優先工作清單開始：每項列出嚴重度、負責單位、期限與原因摘要，點 CTA 按鈕直達對應模組，並直接開啟目標工單或知識文件的詳情抽屜。
3. 檢視營運儀表板六張指標卡：已審互動、自動解決品質、知識缺口、Bot 造成工單、未結失敗案例、發布狀態。
4. 檢視發布門檻速覽：整體品質、引用支撐率、交接召回率、高風險自動回覆；卡上標註分數區間 0–1、1 為滿分。
5. 檢視營運經濟學三張卡；點 `假設參數` 展開可編輯的計算參數。

會看到什麼：

- 經濟學三張卡：每張自動解決工單成本（對比人工成本）、本期 deflection 節省人時、badcase 修復量化回收。區塊頂部與每張卡都標示 seed 模式說明（seed 模式＝以本地示範資料計算的估算值，非生產統計），不聲稱真實成效。
- 假設參數面板可調整：人工單位成本（預設 $24／小時）、平均人工處理分鐘數（預設 12 分鐘）、每張自動解決工單 bot 成本（預設 $0.42）、每件修復 badcase 避免工單數（預設 45）、本期自動解決量（預設由 seed KPI 分群推導，可手動覆寫）。改動即時重算，不會保存。
- 其餘面板：各來源訊號量分布、影響 KPI 的高量案例群、營運隊列摘要。隊列摘要的每張卡可點選，下方隊列詳情顯示該隊列實際的工單數、最高優先級與最近 SLA 工單，點 `前往工單` 直達工單頁並開啟該工單的詳情抽屜。
- 工單相關統計（進行中工單、Bot 造成工單等）會反映你在工單頁做的狀態調整，例如把工單改為 Resolved 後回到儀表板，計數隨之減少。

管理員判斷重點：

- 高風險自動回覆應接近或等於 `0.00`，交接召回率應達到 `1.00`。
- 經濟學卡片用於對上級溝通投資報酬，引用時必須說明是示意值與假設參數。

### 2. 對話審查

**你想完成什麼**：審查 bot 已送出的回答有沒有根據、決策鏈是否合理，並把值得回歸測試的互動轉存為評測案例。

怎麼做：

1. 點擊 `對話審查`，先進入列表層：左側是多來源訊號佇列，右側是緊湊的已送出回覆清單。
2. 訊號佇列用來源 chip 篩選：Web/App Chat、X、LINE、Telegram、Discord、內部通報。點任一筆訊號可展開，看完整原文、來源信任度、重複叢集與地區／語言／產品，並可點 `檢視相關回覆紀錄` 直接進入對應互動的審查層。
3. 或在已送出回覆清單點擊 `檢視回覆與紀錄`，進入全版審查層；頂端顯示麵包屑「對話審查 / 場景標題」與 `返回清單` 按鈕。
4. 在審查層左側的互動紀錄，檢查地區、語言、產品、風險標籤與完整對話串。
5. 點擊訊息內的引用按鈕，右側 Trace 面板下方的引用證據區會顯示對應的 KB chunk 原文。
6. 在回覆 Trace 面板逐節點檢查：來源正規化、意圖、風險、檢索、生成、引用、驗證、交接，每節點有 pass／watch／blocked 狀態。
7. 點擊 `建立評測案例`，把本次互動的訊息、引用、trace 與預期行為打包轉存。
8. 點 `返回清單`（或麵包屑的「對話審查」）回到列表層。

會看到什麼：

- 開啟互動紀錄的動作會寫入稽核紀錄（trace 審查事件）。
- 轉存成功後，互動紀錄下方顯示評測案例 ID。評測案例依情境各自記錄，換一個情境轉存不會覆蓋前一筆；所有已轉存的 ID 會完整列在 `品質與發布 → 評測批次` 的「連結評測案例」卡。

管理員判斷重點：

- 回答是否引用政策來源、引用 chunk 是否真的支撐回答內容。
- 高風險案例是否避免帳戶層級結論，並正確走交接節點。
- trace 是否足以解釋 bot 為什麼這樣回答。

### 3. 知識治理

知識治理有三個分頁：`知識清冊`、`缺口挖掘`、`SOP 管理`。

#### 3.1 知識清冊

**你想完成什麼**：確認 bot 可引用的知識是最新、已索引、能支撐回答的。

怎麼做：

1. 點擊 `知識治理`，預設進入 `知識清冊`。
2. 檢視四個總覽計數磚：已索引文件、需重建索引、已排入索引、chunks 總數。
3. 點知識表格的任一列或列上的 `開啟紀錄`，開啟知識紀錄抽屜：上半部顯示負責單位、語言、生效日、向量索引、檢索設定、最後索引時間與版本紀錄，下半部列出該文件的 chunk 證據。
4. 依需要點抽屜底部的操作按鈕：`查看 chunks`、`審查引用範圍`；需重建索引的文件會多一顆 `排入重建索引`。表格列上的需重建索引文件也有 `重建索引` 按鈕，可不開抽屜直接排入。

會看到什麼：

- 每個操作都有即時的狀態回饋文字。
- 抽屜內的 chunk 證據只顯示目前開啟文件的 chunks；點 `查看 chunks` 會捲動到抽屜下半部的證據區並高亮這些 chunks。從對話審查點過引用時，對應 chunk 也會高亮。
- 點 `排入重建索引` 後，該文件的索引狀態變為「已排入索引」，計數磚即時連動（需重建索引 −1、已排入索引 +1），重新整理後保留，並寫入稽核事件（排入索引）。
- 點 `審查引用範圍` 會在知識紀錄掛上「引用審查中」標記，再點一次可取消。

管理員判斷重點：

- 文件是否為 Published 且已索引；需重建索引的文件會影響回答引用與發布信心。
- 知識表格標題旁的 `檢索政策` 按鈕會開啟彈窗，說明三件事：KB snapshot 是評測與發布門檻使用的版本化狀態、已發布文件必須完成 embedding、只有允許引用的 chunks 可支撐回答。

#### 3.2 缺口挖掘

**你想完成什麼**：從未解決與交接對話自動長出知識庫工作，取代人工盲找缺口。

怎麼做：

1. 切換到 `缺口挖掘` 分頁。
2. 檢視新興問題簇列表：簇名、量體、週趨勢、樣本語句、關聯 KB 狀態（缺漏／薄弱）、目前狀態。
3. 點選一個問題簇開啟詳情抽屜（進入分頁時不會預設選中任何簇），內容包含量體、週趨勢、樣本語句與 AI 起草的 FAQ 候選（草稿問題、草稿答案、引用來源、審核意見）。
4. 在抽屜底部做出審核決定：`採納入庫`、`退回重寫` 或 `標記不適合自動化`。

會看到什麼：

- 每個審核決定都有回饋訊息，並寫入稽核紀錄（FAQ 審核事件，操作者為 Knowledge Owner）。
- 採納後該簇狀態變為 Adopted，並顯示 deflection 成效追蹤：入庫前後由 bot 解決、無需人工交接的比例對照。
- 採納會實際新增一筆 Draft 知識文件（含草稿答案的 chunk），在 `知識清冊` 的表格與計數磚立即可見，重新整理後保留。
- 樣本數低於起草門檻的簇顯示為「觀察中」，不會產生 FAQ 草稿。

管理員判斷重點：

- 完整走一圈是「發現簇 → 審核候選 → 看到成效追蹤」；不是每個簇都該寫 FAQ，退款、補償這類主題應標記不適合自動化。

#### 3.3 SOP 管理

**你想完成什麼**：把 SOP 從文件升級為結構化物件，明確標出每一步 bot 可以做什麼、不能做什麼。

怎麼做：

1. 切換到 `SOP 管理` 分頁。
2. 檢視 SOP 列表：標題、觸發情境、風險等級、負責單位、狀態、步驟數。
3. 點選一筆 SOP 開啟詳情抽屜，內含步驟時間軸：每個步驟有指示內容、分支條件，以及自動化邊界標記。
4. 檢視抽屜內的「對應 chat flow 節點」，確認 SOP 由哪些 flow 節點在 bot trace 中執行。

會看到什麼：

- 三種自動化邊界：`可自動執行`（低風險可逆步驟，如收集非敏感欄位）、`需人工確認`（bot 準備動作、指定負責人確認後才執行，如帳戶凍結）、`禁止自動化`（無論信心多高都由人工處理，如資產移動決策、退款、補償承諾）。
- 含禁止自動化步驟的 SOP 會在步驟上以紅色標記突顯。

管理員判斷重點：

- 邊界標記是治理承諾：檢查高風險 SOP 是否至少有一個禁止自動化步驟，且與 handoff 行為一致。

### 4. 座席輔助

**你想完成什麼**：用座席的實際行為量測 AI 建議回覆與 AI 摘要的品質，決定哪些意圖可以升級為自動送出、哪些該檢討 prompt。這是治理面板，不是回覆台。

怎麼做：

1. 點擊 `座席輔助`。
2. 檢視四張指標卡：建議採納率（直接採納／修改後採納／棄用）、修改幅度分布（無修改／輕度／重度）、AI 摘要重寫率、採納組與棄用組的處理時長差。
3. 檢視政策提示區的規則化建議。
4. 在建議明細表用座席動作 chip 篩選（全部／已採納／已修改／已棄用），點選任一筆開啟建議詳情抽屜。
5. 抽屜內並排比對「AI 建議回覆 vs 座席最終回覆」與「AI 摘要 vs 座席摘要」，並附 trace 參照。
6. 對低品質建議點抽屜底部的 `標記為 badcase`。

會看到什麼：

- 修改幅度分布卡的主值是最大類別的佔比（例如「67% 無修改」），下方明細列出無修改／輕度／重度各自的筆數。
- 政策提示的規則：某隊列採納率（直接＋修改後）超過 80% 且屬低風險隊列（General Support、Knowledge Ops）時，提示「可評估開放自動送出」；棄用率超過 50% 時，提示「檢討 prompt 或暫時關閉此隊列的建議」。
- 標記 badcase 後會出現確認訊息，該筆案例流入 `品質與發布 → 失敗案例` 清單（負責單位 PM、狀態 Open），同時寫入稽核紀錄。同一筆建議不能重複標記。

管理員判斷重點：

- 指標卡會依門檻自動變為 watch 狀態（例如棄用率超過 30%、摘要重寫率超過 25%、處理時長差超過 2 分鐘），優先處理 watch 卡對應的隊列。
- 升級自動送出前，採納率、風險等級、修改幅度三個條件要同時滿足。

### 5. 工單與交接

**你想完成什麼**：追蹤 bot 建立的工單消耗了多少人工容量，並確認高風險案例的交接包品質。

怎麼做：

1. 點擊 `工單與交接`。
2. 檢視工單表格：問題摘要、隊列、優先級、SLA、處理狀態、負責人（PIC）。
3. 直接在表格內用下拉選單調整處理狀態（Open／Pending review／Escalated／Resolved）與負責人（PIC）；點表格列或 `開啟詳情` 可開啟工單詳情抽屜。
4. 工單詳情抽屜包含來源訊號、trace 參照、AI 案例摘要、下一步與處理時間線。
5. 檢視右側隊列狀態摘要：高優先數、已升級數、SLA 關注數。
6. 在工單詳情抽屜下半部檢視人工交接預覽：觸發原因、AI 摘要、必要欄位清單、風險警示。頁面上沒有獨立的交接預覽區塊。

會看到什麼：

- 狀態或負責人（PIC）調整後有即時回饋訊息，變更會保存在本機，儀表板的工單統計同步反映。
- 交接預覽對應目前開啟的工單；若該工單沒有 bot 產生的交接包（例如低風險自動解決），抽屜內顯示「此工單無交接包」的空狀態。

管理員判斷重點：

- 高風險安全案例是否進入 `Security-L2`，法遵或知識缺口是否分派給正確負責單位。
- 交接包必要欄位是否齊全、風險警語是否禁止 bot 做帳戶操作或安全結論。
- 下一步是否具體，且不要求 bot 做帳戶層級決策。

### 6. 品質與發布

品質與發布有五個分頁：`發布門檻`、`KPI`、`評測批次`、`失敗案例`、`稽核`。頁面頂端固定顯示發布狀態摘要（阻擋套件數、失敗門檻數、候選版本退化數），由目前的門檻結果與評測資料即時計算，不是靜態文案。

#### 6.1 發布門檻

**你想完成什麼**：先看清楚候選版本改了什麼，再依 eval 證據做出可稽核的發布決策。

怎麼做：

1. 點擊 `品質與發布`，預設進入 `發布門檻`。
2. 每個發布套件卡片顯示狀態、prompt／KB snapshot／檢索設定版本與門檻結果。
3. 點 `檢視變更` 展開 flow version diff：
   - 唯讀流程拓撲圖，每個節點標示新增／移除／修改／未變更；
   - 節點變更清單與說明；
   - prompt 段落修改前後對照；
   - KB snapshot 與檢索設定異動。
4. 檢查決策原因欄：系統已預填一行 diff 摘要，可補上門檻依據、審查者或 rollback 條件。
5. 做出決策：`推進`、`阻擋發布` 或 `請求審查`。

會看到什麼：

- 有阻擋門檻的套件會列出失敗原因，且 `推進` 按鈕不可用，直到門檻修正。
- 決策後卡片內出現決策橫幅（決策結果與理由），並寫入稽核紀錄。

管理員判斷重點：

- 決策順序：先看 diff 確認改了什麼 → 再看 eval 對照與退化數 → 才做決策。理由欄是稽核回溯的依據，不要留空泛描述。

#### 6.2 KPI

**你想完成什麼**：定期檢視 bot 對客服營運的實際影響，找出該優先改善的分群。

怎麼做：

1. 切換到 `KPI` 分頁。
2. 檢查上方 KPI 卡：自動解決、人工交接、重複進線、引用失敗、SLA 風險，各含目標值、趨勢與洞察。
3. 檢視右側 KPI 關注清單：列出狀態為 risk／watch 的指標，點選會定位並突顯對應的 KPI 卡。
4. 在分群分析表依來源與案例類型下鑽，閱讀每個分群的 review focus；點列上的 `檢視對話` 直達對話審查，來源篩選已套用該分群的管道。

管理員判斷重點：

- 自動解決率提升不能犧牲高風險安全性；重複進線高代表回答不夠清楚；引用失敗暴露知識庫缺口。

#### 6.3 評測批次

**你想完成什麼**：用同一批評測案例比較不同發布設定，並確認 LLM judge 本身可信，評測結果才有資格支撐發布。

怎麼做：

1. 切換到 `評測批次` 分頁。
2. 點 `啟動評測`：按鈕先進入「評測中…」的執行過渡（約 1.7 秒，期間不可重複啟動），完成後評測執行狀態變為「已完成」。
3. 在批次表比較各 run 的整體分、引用、交接召回、自動回覆與退化數；點選任一批次看詳情（資料集與 flow／prompt／KB／檢索版本組合）。分數區間 0–1、1 為滿分，頁面上有標註。
4. 檢視 Judge 校準面板：每個 judge 版本顯示與人工一致率（對照門檻）、抽樣案例數、抽樣複核佇列、換版漂移警示。
5. 點 `匯出 CSV`，在預覽彈窗確認欄位後點 `下載 CSV`，下載 `botops-eval-summary.csv`。

會看到什麼：

- 評測完成後，評測執行狀態面板顯示結果摘要：候選版本與基準版本的整體分數對照與差值。
- **待人工複核規則**：當某 judge 版本的與人工一致率低於門檻，該版本卡片標示「低於門檻」並顯示警示「評測結果待人工複核，不得作為發布依據」；批次表中所有使用該 judge 版本的 run 也會掛上「待人工複核」標記。
- CSV 匯出僅包含 run-level 指標，預覽彈窗明確聲明不含客戶識別資訊、私人原文訊息或帳戶資料。
- 啟動評測、評測完成與 CSV 匯出都寫入稽核紀錄。

管理員判斷重點：

- 有「待人工複核」標記的批次分數不能拿來做 Promote 依據；先清抽樣複核佇列、確認 judge 校準恢復。

#### 6.4 失敗案例

**你想完成什麼**：把失敗案例轉為有負責單位、有重測指標的修正任務，並追蹤到修好為止。

怎麼做：

1. 切換到 `失敗案例` 分頁。
2. 檢視列表的失敗標籤與狀態，用下拉選單調整狀態：`Open`、`In review`、`Fixed`。
3. 點 `開啟紀錄` 看詳情：低分項目、追蹤診斷、需調整節點、調整方式、重測指標、負責單位。

會看到什麼：

- 失敗案例有兩個來源：評測低分 rows，以及座席輔助標記的低品質建議（ID 以 `badcase_assist_` 開頭，重測指標為建議採納率）。

管理員判斷重點：

- 先定位問題屬於 prompt、檢索、知識、風險路由還是交接，再確認負責單位與重測指標是否明確。

#### 6.5 稽核

**你想完成什麼**：確認重要治理動作皆已留存紀錄，可回溯操作者、對象與時間。

怎麼做：

1. 切換到 `稽核` 分頁。
2. 最新事件顯示在列表最上方；用事件類型 chip 篩選。
3. 點選任一事件，右側顯示事件類型、操作者、實體參照與建立時間。

會看到什麼：

- 稽核事件雙語記錄：zh-TW 模式下事件標題與詳情顯示全中文；切回 EN 顯示英文。切換語言前留下的舊格式事件維持原文。

管理員判斷重點：

- 剛完成的操作（trace 審查、轉存評測案例、FAQ 審核、發布決策等）應立即出現在列表最上方。
- 篩選後事件數與內容應與實際操作一致。

## Audit log 記錄哪些操作

稽核紀錄涵蓋以下九種事件（對照 `src/types.ts` 的 `AuditEventType`）：

| 事件類型 | 觸發操作 | 觸發位置 | 操作者 |
| --- | --- | --- | --- |
| `live_trace_review` | 開啟已回覆互動的回覆與 trace | 對話審查 → 檢視回覆與紀錄 | PM |
| `eval_case_saved` | 將互動紀錄轉存為評測案例 | 對話審查 → 建立評測案例 | PM |
| `eval_runner_started` | 啟動離線評測 | 品質與發布 → 評測批次 → 啟動評測 | Bot Ops |
| `eval_runner_completed` | 評測執行完成（含初始資料同步） | 系統自動記錄 | System |
| `csv_exported` | 下載評測摘要 CSV | 品質與發布 → 評測批次 → 匯出 CSV | PM |
| `release_decision` | 推進／阻擋發布／請求審查（含決策理由） | 品質與發布 → 發布門檻 | PM |
| `faq_candidate_reviewed` | 採納入庫／退回重寫／標記不適合自動化 | 知識治理 → 缺口挖掘 | Knowledge Owner |
| `assist_badcase_flagged` | 將低品質建議標記為 badcase | 座席輔助 → 建議詳情 | PM |
| `knowledge_index_queued` | 將文件排入重建索引 | 知識治理 → 知識清冊 → 排入重建索引 | Knowledge Owner |

每筆事件記錄操作者、事件標題、詳情、實體參照（如情境 ID、評測案例 ID、發布套件 ID）與時間。

## 目前限制

- 本機開發版本使用 seed data，不連接真實 channel、ticket system 或 LLM。
- localStorage 僅供本機流程驗證（語言選擇、稽核事件、評測案例 ID、工單狀態與負責人調整、知識索引狀態、引用審查標記、FAQ 審核結果與採納入庫的知識文件），不是正式資料儲存。
- 儀表板經濟學卡片全部為 seed 模式示意值；假設參數改動即時重算但不保存。
- CSV export 是前端產生，未來應由後端提供 signed export 或 report job。
- Domain pack 只存在於資料層，UI 沒有 pack 切換器；換產業需新增 pack 檔並重新 build。
- 後端 API、資料表與 eval runner job flow 請參考 `docs/backend-api-contract.md`；三層資訊架構請參考 `docs/ia-inventory.html`。
